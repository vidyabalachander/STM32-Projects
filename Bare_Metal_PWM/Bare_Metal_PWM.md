# STM32F407 PWM Bare-Metal Implementation

Fading the onboard LED on the STM32F407G-DISC1 board up using hardware PWM generated by **TIM4 Channel 1**, with the duty cycle stepped directly through register access.

## Overview

This project builds on the earlier Blinky project by replacing a simple GPIO toggle with a PWM signal. Instead of turning the LED fully on or off, TIM4 generates a continuous PWM waveform on the LED pin, and the firmware ramps the duty cycle from 0 to 255 using bare-metal register manipulation to create a smooth fade effect.

## Hardware

- **Board:** STM32F407G-DISC1
- **LED pin:** Connected to TIM4 Channel 1
- **Timer used:** TIM4 (APB1 timer)

## Peripheral Configuration

Generated via CubeMX, TIM4 is configured as follows:

| Setting | Value |
|---|---|
| Prescaler (PSC) | 0 |
| Auto-Reload (ARR / Period) | 255 |
| Counter Mode | Up |
| Auto-Reload Preload | Enabled |
| Channel 1 Mode | PWM Mode 1 |
| Channel 1 Polarity | Active High |
| Initial Pulse (CCR1) | 0 |

## Code Walkthrough

### 1. Starting the PWM channel

PWM generation must be explicitly started after peripheral init, since `MX_TIM4_Init()` only configures the timer — it doesn't start output:

```c
HAL_TIM_PWM_Start(&htim4, TIM_CHANNEL_1);
```

### 2. Increasing the duty cycle = Increasing brightness

Inside the infinite loop, the duty cycle is stepped from 0 to 255 by writing directly to the **Capture/Compare Register (CCR1)**:

```c
for (int i = 0; i <= 255; i++) {
    TIM4->CCR1 = i;
    // change the duty cycle by manipulating the register directly
    // set the value of the CCR1 register to our control number i
    HAL_Delay(10);
}
```

Since `ARR = 255`, `CCR1` can range from 0 (0% duty / LED off) to 255 (~100% duty / LED fully on), giving 256 discrete brightness steps. Each step holds for 10 ms via `HAL_Delay(10)`, producing a full fade cycle (dark → bright) in about 2.5 seconds.

Writing `CCR1` directly (register access) instead of calling a HAL setter function keeps this loop lightweight and makes the PWM/duty-cycle relationship explicit rather than hidden behind an abstraction.

## Key Concepts

- **PWM duty cycle vs. perceived brightness:** Because of how the human eye perceives light, a linear duty-cycle sweep (0→255) does not look linear — brightness appears to change quickly at low duty cycles and barely at all near full brightness. A gamma-corrected lookup table can be applied to `CCR1` values for a more visually linear fade.

## Possible Extensions

- Add gamma correction for a perceptually linear fade.
- Replace the blocking `HAL_Delay` loop with a non-blocking, timer-interrupt-driven fade.
